# PriceAI Umami 数据监测与采集分层规划

生成时间：2026-07-14  
状态：规划完成，待实施  
关联模块：卡网订阅、商家来源、后台管理、数据归因、采集健康、链动小铺采集调度  
关联文档：[`docs/analytics.md`](../../../../analytics.md)、[数据归因与后台分析专区产品规划](../../done/product/2026-07-05_priceai-attribution-analytics-product-plan.md)、[渠道质量评分与动态采集频率规划](../../done/product/2026-06-09_channel-quality-score-and-dynamic-collection-plan.md)、[后台管理左侧导航与工作流拆分规划](2026-07-12_admin-console-left-nav-workflow-split-plan.md)

## 0. 核心结论

本规划把 PriceAI 的数据监测拆成三层：

```text
Umami = 用户行为热度层
Supabase / 采集日志 = 采集健康与业务事实层
PriceAI 后台 = 聚合、解释、分层建议与人工决策层
```

短期目标不是立刻自动改采集调度，而是先把后台数据监测做起来：

1. 用 Umami 作为统一的用户行为统计来源，集中看页面访问、商品详情打开、商家点击、购买跳转、平台筛选和搜索结果点击。
2. 用现有业务库和采集日志判断来源是否采集成功、多久没刷新、失败原因是什么、是否疑似风控。
3. 后台把“用户热度”和“采集健康”放到同一个工作台里，优先暴露高热度但数据陈旧、高热度但采集失败、长期低热度且低价值、长期失败等对象。
4. 采集频率先做建议，不直接自动加频、降频、停用或下架；所有影响线上采集策略的动作先由管理员确认。

## 1. 产品一句话

这是一个面向 PriceAI 运营者的后台数据监测工作台，帮助运营者基于 Umami 用户行为热度和采集健康状态，识别哪些商品、商家和来源应该优先刷新、降频观察、人工复查或进入候补队列，从而提升采集效率和数据新鲜度。

## 2. 背景与问题

### 2.1 当前触发背景

近期采集侧连续出现几个问题：

1. 部分线上采集节点被风控或失败，导致店铺更新时间明显滞后。
2. 店铺数量上来以后，如果所有店铺都按同一频率采集，会浪费服务器、代理和目标站请求预算。
3. 有些店铺或商品真实用户关注度高，但刷新不够及时；有些长期排在后面、没人点、价格也不优，却仍然占用同样采集资源。
4. 后台现在更像“结果查看”和“人工操作”，缺少一个能把用户行为、采集健康和运营动作串起来的数据监测层。

### 2.2 既有规划结论

已有规划提供了三个重要前提：

1. [渠道质量评分与动态采集频率规划](../../done/product/2026-06-09_channel-quality-score-and-dynamic-collection-plan.md)提出过 S/A/B/C/D 分层和动态采集频率，但后续复盘结论是不继续用单一总分直接驱动采集和展示，应把价格、采集稳定性、用户反馈和商业归因拆成独立信号。
2. [数据归因与后台分析专区产品规划](../../done/product/2026-07-05_priceai-attribution-analytics-product-plan.md)已经确认前台不展示点击数，后台才做运营判断；点击热度不能被包装成可信度或成交量。
3. [`docs/analytics.md`](../../../../analytics.md)已经记录了 PriceAI 的自托管 Umami 接入，Umami 适合做页面访问、来源、设备、地域和关键点击的日常快速观察。

本规划在这些结论上做一次聚合：Umami 作为用户行为热度统一来源，业务库继续承载采集事实，后台只负责聚合、解释和给出运营建议。

## 3. 产品边界

### 3.1 本版本做什么

1. 建立后台「数据监测 / 采集优先级」视角，把 Umami 行为数据和采集健康数据放在同一个工作流里看。
2. 统一关键行为口径：商品详情打开、商家入口点击、购买外链点击、平台页 CTA、筛选交互、搜索结果点击。
3. 形成商品、商家 / 来源、平台、页面四个维度的 7 天 / 30 天热度摘要。
4. 标记高热度但数据陈旧、高热度但采集失败、低热度低价值、长期失败和疑似风控来源。
5. 输出采集频率建议和复查队列，但不自动修改线上调度。

### 3.2 本版本不做什么

1. 不把 Umami 当作采集是否成功、库存是否真实、价格是否可信的事实来源。
2. 不在前台展示精确访问量、点击量、CTR 或“热门商家”标签。
3. 不用点击量直接决定前台排序、商家可信度、商家下架或赞助展示。
4. 不把一次高点击视为成交，也不把低点击视为一定没有价值。
5. 不在第一阶段做全自动动态调度；先做数据可见、建议和人工确认。

## 4. 产品形态判断

| 候选形态 | 是否采用 | 原因 |
| --- | --- | --- |
| Web 后台工作台 | 第一版采用 | PriceAI 已有后台管理，适合展示表格、筛选、异常队列和运营动作 |
| H5 | 不采用 | 数据密度高，移动端不适合做采集调度和异常排查 |
| App | 不采用 | 没有独立分发价值 |
| 小程序 | 不采用 | 内部运营工具不需要 |
| 浏览器插件 | 暂不采用 | 可能适合未来人工核验目标站，但不是本阶段重点 |
| Bot / 对话助手 | 后续可作为告警入口 | 可用于每日摘要或异常提醒，但不能替代后台判断 |
| API / MCP / 插件 | 后续可考虑 | 当数据口径稳定后，可为自动化运维或日报生成提供接口 |

## 5. 用户、场景与任务

| 用户 | 场景 | 触发点 | 期望结果 | 产品需要支持 |
| --- | --- | --- | --- | --- |
| PriceAI 运营者 | 每天查看采集是否正常 | 用户发现更新时间很久以前，或后台出现大量失败 | 快速知道是整体失败、节点失败、来源失败还是个别风控 | 数据总览、异常队列、来源明细、失败原因聚合 |
| PriceAI 运营者 | 判断哪些店铺值得更高频刷新 | 店铺数量增加，服务器和代理资源有限 | 把高热度、高点击、高价值来源优先放进高频队列 | Umami 热度排行、价格价值辅助、采集健康联动 |
| PriceAI 运营者 | 判断哪些店铺可以降频或候补 | 部分店铺长期低点击、低价格竞争力、还频繁失败 | 降低低价值来源对采集资源的占用 | 低热度候选、低价命中、长期失败、人工备注 |
| PriceAI 运营者 | 排查采集节点和代理是否失效 | 某一批店铺突然全失败或更新停滞 | 确认问题集中在哪个节点、来源类型、失败原因 | 节点健康、最近运行、失败分布、时间窗口对比 |
| 产品负责人 | 做后续分层采集策略 | 想把“该拉高拉高，该删除删除”变成可执行机制 | 有一套可解释、可回滚、可人工确认的分层建议 | 分层规则、建议队列、执行记录、观察指标 |

## 6. 核心用户旅程

```text
进入后台
-> 打开数据监测总览
-> 查看最近 30 / 60 / 120 分钟采集新鲜度
-> 查看近 7 / 30 天用户行为热度
-> 进入异常队列
-> 识别高热度陈旧 / 高热度失败 / 长期低热度 / 连续风控来源
-> 查看来源详情和历史趋势
-> 生成采集频率建议
-> 人工确认处理动作
-> 观察下一轮采集和用户行为变化
```

| 步骤 | 用户动作 | 系统动作 | 用户获得什么 | 关键风险 |
| --- | --- | --- | --- | --- |
| 1 | 进入后台数据监测 | 汇总 Umami 行为数据和采集健康数据 | 全局是否正常的第一判断 | Umami 或采集日志延迟造成误判 |
| 2 | 查看异常队列 | 按热度、陈旧程度、失败次数排序 | 当前最值得处理的问题列表 | 排序过度依赖单一指标 |
| 3 | 进入来源 / 商品详情 | 展示热度趋势、最近采集、报价情况、失败原因 | 判断是否加频、降频或复查 | 缺少上下文导致误停优质来源 |
| 4 | 查看分层建议 | 给出高频、常规、低频、复查、候补建议 | 把运营判断转成动作候选 | 自动化过强会误伤 |
| 5 | 人工确认动作 | 记录处理动作和原因 | 可追溯、可回滚的运营决策 | 没有执行记录会反复踩坑 |
| 6 | 复盘效果 | 对比执行前后的刷新率、失败率和用户点击 | 判断规则是否有效 | 没有复盘就无法调优 |

## 7. 信号分层

### 7.1 用户行为热度层：Umami

Umami 负责回答“用户实际在看什么、点什么、从哪里进入”。

建议优先统一这些行为：

| 行为 | 用途 | 关键维度 |
| --- | --- | --- |
| 商品详情打开 | 判断哪些标准商品真实有人看 | `product_id`、平台、页面路径 |
| 商家 / 来源入口点击 | 判断哪些店铺真实有人进入 | `source_id`、平台、商品 |
| 购买外链点击 | 判断用户是否有明确交易意图 | `source_id`、`product_id`、入口位置 |
| 平台页 CTA 点击 | 判断 SEO / 内容页是否把用户带到工具 | 平台、目标页面、CTA 位置 |
| 筛选交互 | 判断用户常用筛选条件 | 平台、分类、排序、库存状态 |
| 搜索结果点击 | 判断搜索后的真实目标 | 标准商品、平台、结果位置 |

注意：不要把完整搜索词、原始商品标题、渠道链接、邮箱、联系方式等敏感信息发给 Umami。

### 7.2 采集健康层：Supabase / 采集日志

采集健康层负责回答“数据是不是真的新、采集有没有成功、失败在哪里”。

核心信号：

| 信号 | 用途 |
| --- | --- |
| 最近成功采集时间 | 判断是否陈旧 |
| 最近检查时间 | 区分未运行和运行失败 |
| 最近 N 次成功率 | 判断稳定性 |
| 连续失败次数 | 判断是否需要降频或复查 |
| 失败类型 | 区分风控、超时、HTTP 错误、解析失败、无报价 |
| 节点与代理 | 判断是否是机器 / IP / 区域问题 |
| 有效报价数 | 判断来源是否仍有业务价值 |
| 下架 / 无货比例 | 判断是否占用无效采集资源 |

### 7.3 业务价值层：价格与覆盖

业务价值层负责回答“这个来源即使有人看，是否值得高频采集”。

核心信号：

| 信号 | 用途 |
| --- | --- |
| 低价命中 | 是否经常进入同标准商品低价前列 |
| 覆盖商品数 | 是否只靠单个爆品 |
| 有货报价数 | 是否仍有有效供给 |
| 平台覆盖 | 是否覆盖 ChatGPT、Claude、Gemini 等核心平台 |
| 资料核验 / 稳定性反馈 | 是否有人工或登录用户反馈支持 |

### 7.4 分层原则

分层不是一个统一总分，而是把多个独立信号组合成可解释建议。

| 情况 | 建议动作 |
| --- | --- |
| 高热度 + 价格有优势 + 采集稳定 | 提高采集频率，优先保障 15-30 分钟内刷新 |
| 高热度 + 数据陈旧 | 进入紧急刷新队列，优先排查对应节点 |
| 高热度 + 连续失败 / 疑似风控 | 进入风控复查队列，不盲目加大并发 |
| 中热度 + 采集稳定 | 保持常规频率 |
| 低热度 + 低价命中少 + 商品少 | 降低采集频率，减少资源占用 |
| 长期低热度 + 长期失败 | 进入候补 / 暂停候选，需要人工确认 |
| 低热度但价格极优 | 不直接降频，保留常规或专项观察 |

| 采集健康联动页 | 排查采集失败和陈旧 | 节点状态、失败原因、30 / 60 / 120 分钟覆盖率 | 正常、节点异常、风控集中 |
| 分层建议队列 | 处理加频、降频、复查候选 | 高频、常规、低频、风控复查、候补列表 | 待确认、已处理、已忽略 |
| 来源详情抽屉 | 看单个来源的上下文 | 热度趋势、报价数量、低价命中、失败历史、人工备注 | 正常、数据不足 |
| 复盘记录页 | 回看调度动作效果 | 操作记录、前后指标、备注 | 空、正常 |

## 11. 关键状态与边界

| 状态 | 出现时机 | 产品应该怎么处理 |
| --- | --- | --- |
| Umami 查询失败 | 自托管 Umami 不可用或认证失败 | 后台展示明确失败，不影响采集健康数据；不能生成热度建议 |
| Umami 数据延迟 | 最近事件时间落后明显 | 标注数据延迟，避免把低热度误判为没人看 |
| 事件口径缺失 | 某些页面没有稳定 `product_id` 或 `source_id` | 标为“不可归因”，进入事件补齐清单 |
| 样本过少 | 新功能、新商品或低流量页面 | 不给强建议，只展示观察中 |
| 高热度陈旧 | 用户点击高但报价超过目标新鲜度 | 进入紧急刷新候选 |
| 高热度失败 | 用户点击高但来源连续失败 | 进入风控复查，优先判断是否需要换节点 / 代理 |
| 低热度低价值 | 长期没人点、低价命中少、报价少 | 进入降频候选，不自动停用 |
| 长期失败 | 连续失败且无有效报价 | 进入候补 / 暂停候选，必须人工确认 |
| 异常点击 | 短时间单来源点击异常高 | 标记异常，不直接提高采集频率 |

## 12. 成功标准

| 指标 | 说明 | 第一版目标 |
| --- | --- | --- |
| 监测可见性 | 后台能同时看到行为热度和采集健康 | P0 上线后运营不再只靠截图和手工查库判断 |
| 高热度陈旧识别 | 能列出高热度但超过目标新鲜度的商品 / 来源 | 第一版至少覆盖卡网订阅核心来源 |
| 失败原因聚合 | 能看出失败集中在来源、节点、风控还是解析 | 第一版能支持紧急排查 |
| 事件归因覆盖 | 关键点击事件能落到 `product_id` / `source_id` 等稳定维度 | 第一版覆盖商品详情和购买跳转主链路 |
| 分层建议可解释 | 每条建议能说明为什么建议加频、降频或复查 | 不出现黑盒总分 |

## 13. 分阶段落地路线

### P0：监测口径与后台只读总览

目标：

1. 确认 Umami 事件口径和稳定维度。
2. 后台能查询近 7 / 30 天关键行为数据。
3. 后台能展示采集新鲜度、失败原因和最近运行状态。
4. 输出高热度陈旧、高热度失败、长期失败三个只读列表。

验收：

1. 能从后台看到商品和来源的行为热度排行。
2. 能从后台看到 30 / 60 / 120 分钟采集覆盖率。
3. 能识别“用户在点，但没刷新”的来源。
4. 不改变任何线上采集调度。

### P1：分层建议队列

目标：

1. 生成高频、常规、低频、风控复查、候补五类建议。
2. 每条建议展示独立信号：热度、价格价值、采集健康、失败原因。
3. 管理员可以确认、忽略或备注。

验收：

1. 每条建议都能解释原因。
2. 管理员可以按建议批量处理，但默认不自动执行。
3. 降频、暂停和候补必须人工确认。

### P2：运营复盘与告警

目标：

1. 记录每次人工调度动作。
2. 对比动作前后的刷新率、失败率和点击趋势。
3. 提供每日 / 每周异常摘要。

验收：

1. 能回答“上次为什么把这个店铺降频 / 恢复 / 加频”。
2. 能判断调度动作是否改善了新鲜度和失败率。
3. 能在节点整体异常时及时发现。

### P3：半自动调度试运行

目标：

1. 对低风险动作做半自动试运行，例如高热度稳定来源进入高频队列。
2. 对高风险动作保持人工确认，例如暂停、删除、长期候补。
3. 规则可回滚，可灰度到部分来源。

验收：

1. 高频来源的新鲜度提升。
2. 低价值来源占用的采集资源下降。
3. 风控失败没有因为盲目并发而扩大。

## 14. 与既有规划的关系

### 14.1 与数据归因规划

原数据归因规划强调自有事件表适合作为可 join、可审计的业务账本。这个判断仍然成立，尤其适合未来商家报表、赞助复盘、对外证明和严格去重。

本规划的调整是：对于当前内部运营和采集分层，优先复用自托管 Umami 作为统一行为热度来源，后台通过 Umami API 查询和汇总。这样可以更快建立监测闭环，减少重复埋点和重复统计面板。

建议边界：

```text
内部热度观察、趋势判断、采集优先级建议：优先 Umami
采集成功、报价事实、失败原因、库存状态：业务库 / 采集日志
商家报表、赞助复盘、严格审计：保留自有归因能力
```

### 14.2 与渠道质量评分规划

旧规划里的 S/A/B/C/D 总分不应直接作为单一权威评分。本规划保留“分层采集”的方向，但把分层改成多信号建议：

1. 行为热度来自 Umami。
2. 采集稳定性来自采集日志。
3. 价格价值来自报价和低价命中。
4. 用户 / 人工反馈来自资料核验和稳定性反馈规划。

后台展示这些独立信号，再给出建议动作。

### 14.3 与资料核验商家和稳定性反馈规划

用户反馈、资料核验、稳定性样本可以增强商家质量判断，但不替代行为热度。

建议后续关系：

1. 用户行为告诉我们“用户关注什么”。
2. 资料核验和稳定性反馈告诉我们“这个来源是否更可信”。
3. 采集健康告诉我们“这个来源现在能不能稳定刷新”。
4. 价格与库存告诉我们“这个来源是否有实际供给价值”。

## 15. 风险与约束

| 风险 | 影响 | 处理方式 |
| --- | --- | --- |
| 点击被误读为信任背书 | 误导前台排序和用户判断 | 点击数据只在后台使用，不公开精确数字 |
| Umami 事件缺稳定 ID | 无法按商品 / 来源聚合 | P0 先做事件口径审计，缺口补齐后再进入分层 |
| Umami API 不稳定 | 后台热度数据不可用 | 展示降级状态，采集健康仍可独立查看 |
| 异常点击或刷量 | 错误提高采集频率 | 标记异常趋势，不把单一点击量作为动作依据 |
| 自动调度误伤 | 优质来源被降频或停用 | P0-P2 只做建议，P3 才小范围半自动 |
| 高并发扩大风控 | 更多节点被封控 | 高热度失败进入风控复查，不盲目提高并发 |

## 18. 建议下一步

建议按下面顺序推进：

1. 先做 P0 事件口径审计：确认当前 Umami 事件是否有稳定 `product_id`、`source_id`、平台和入口位置。
2. 再做后台只读监测：先把近 7 / 30 天商品热度、来源热度、30 / 60 / 120 分钟新鲜度和失败原因放在一起。
3. 第三步做异常队列：优先输出高热度陈旧、高热度失败、长期失败三个列表。
4. 数据观察 7-14 天后，再进入 P1 分层建议队列。
5. 等建议命中率稳定后，才考虑 P3 半自动调度。
