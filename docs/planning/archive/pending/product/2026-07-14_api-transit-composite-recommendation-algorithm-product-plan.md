# PriceAI 中转 API 综合推荐算法产品规划

> 文档类型：产品规划 / PRD Lite / 排序与质量信号方向
> 文档版本：V0.1
> 更新时间：2026-07-14
> 当前阶段：当前版已上线，目标版待后续确认和分阶段实施
> 生产基线：commit `5fb509b`，Cloudflare Actions `29267043533`

## 1. 产品一句话

中转 API 综合推荐是 PriceAI 面向模型 API 用户的购前决策能力：围绕用户想调用的具体模型和可选分组线路，综合比较模型质量、价格、响应速度、稳定性和缓存效率，帮助用户筛出值得小额试用的线路，而不是给整个站点做无法解释的单一背书。

## 2. 背景与用户问题

用户选择中转 API 时，真正想完成的任务不是找到“全网第一站点”，而是回答以下问题：

1. 这个分组提供的模型是不是真的，是否存在降质、暗调路由或能力缺失。
2. 调用目标模型的实际价格是否足够低。
3. 首 Token 是否快，后续输出是否流畅。
4. 最近是否稳定，短期故障和长期表现分别如何。
5. 长上下文场景下，缓存能否真正覆盖大量重复输入。
6. 当前结论来自多少公开样本，数据是否足够新、足够可信。

当前中转站市场的主要困难是：同一站点包含多个模型、分组、号池和渠道，不同线路的价格、缓存、成功率和延迟可能明显不同。若先把站点所有数据汇总成一个分数，再进行排序，很容易把不同线路的优势拼接在一起，形成用户实际上无法购买或选择的“虚拟最优线路”。

## 3. 当前生产基线

2026-07-14 已上线第一版中立综合排序，主要解决商业关系直接影响自然排序的问题。

### 3.1 当前权重

| 当前指标 | 权重 | 当前状态 |
| --- | ---: | --- |
| 成本 | 30% | 已生效 |
| 公开近 60 次成功率 | 18% | 已生效 |
| 公开近 7 日成功率 | 12% | 已生效 |
| TTFT | 15% | 权重已预留，暂无正式公开字段，当前不产生排序差异 |
| 缓存命中率 | 10% | 已生效 |
| 模型真假检测 | 15% | 已预留，随公开检测报告逐步生效 |

### 3.2 当前准入规则

- 没有可比较倍率的站点不进入综合推荐。
- 没有公开稳定性样本的站点不进入综合推荐。
- 公开监测超过 7 天未更新的站点不进入综合推荐。
- 商业关系、站点系统、资料完整度和主观风险标签不直接进入性能分。

### 3.3 当前版本仍需继续解决的问题

- 排序对象仍偏站点级，成本、稳定性、缓存和模型检测可能不是同一条线路。
- 相同公开监测事件可能被挂到多个模型或分组记录，需要建立可靠的事件去重规则。
- 缓存目前使用筛选范围内代表值，后续应严格跟随具体分组线路。
- TTFT、TPS 和模型质量数据尚未形成稳定公开口径。
- 缺失指标不能长期等同于真实低分，需要同时表达表现与证据覆盖度。
- “全部”页面最终采用单一总榜还是场景推荐，暂不在本文直接拍板。

## 4. 产品目标与边界

### 4.1 本规划做什么

1. 明确综合推荐的底层评分对象和目标权重。
2. 统一模型质量、价格、速度、稳定性和缓存的产品含义。
3. 规划暂缺 TTFT、TPS 和模型检测数据时的公平处理方式。
4. 规划具体模型、分组线路、模型家族和站点之间的汇总关系。
5. 让用户能看懂“为什么推荐”，而不是只看到一个无法复核的总分。

### 4.2 本规划不做什么

1. 不在当前阶段决定“全部”页面必须保留唯一第一名。
2. 不把赞助、AFF、入驻或合作关系加入自然推荐分。
3. 不把站点系统、来源渠道或资料填写多少直接视为性能优势。
4. 不把最低充值、退款规则和运营时间混入技术性能分；这些继续作为风险提示、筛选项或特定场景参考。
5. 不要求当前立刻实现全部目标指标，先保留可渐进接入的算法结构。

## 6. 核心产品决策

### 6.1 底层评分对象

综合推荐的最小评分对象确定为：

```text
站点 × 标准模型 × 分组 / 线路
```

每条线路分别持有：

- 对应模型的实际成本。
- 去重后的近 60 次与近 7 日成功率。
- TTFT、尾部延迟和输出速度。
- 当前分组的缓存命中率。
- 当前模型和分组的质量检测结果。
- 数据更新时间、样本量和证据覆盖度。

站点级、模型家族级和“全部”页面的结果，都由线路分向上汇总，不再先把站点所有模型混成一个分数。

### 6.2 目标权重

| 一级指标 | 目标权重 | 用户问题 |
| --- | ---: | --- |
| 模型质量 | 25% | 模型是否真实，能力是否完整，是否存在降质 |
| 价格 | 25% | 目标模型在该线路上的实际成本是否合理 |
| 响应速度 | 20% | 首 Token 是否快，后续输出是否流畅 |
| 稳定性 | 20% | 短期和长期是否持续可用 |
| 缓存效率 | 10% | 长上下文场景中缓存读取覆盖是否充分 |
| 合计 | 100% | |

目标公式：

```text
综合推荐分
= 模型质量分 × 25%
+ 价格分 × 25%
+ 响应速度分 × 20%
+ 稳定性分 × 20%
+ 缓存效率分 × 10%
```

该权重结构体现三个核心判断：

- 模型质量和价格是用户选择线路时权重最高的两项。
- 响应速度直接影响交互体感，需要作为高权重独立维度。
- 稳定性通过权重与推荐门槛共同保障，避免低价但频繁失败的线路进入强推荐。

## 7. 缺失数据与证据置信度

### 7.1 中性先验

暂缺 TTFT、TPS 或模型检测时，不直接按 0 分处理，也不因为缺失数据而获得额外优势。

每项指标使用：

```text
调整后指标
= 50
+ 数据置信度 ×（实际指标 - 50）
```

含义：

- 没有数据：按中性 50 分，不产生相对排序差异。
- 样本较少或数据较旧：结果向 50 分收缩。
- 样本充分且数据新鲜：接近实际指标。
- 明确检测为假模型或错误路由：不使用中性先验，直接触发质量退出规则。

### 7.2 证据覆盖度

综合分之外，独立计算：

```text
证据覆盖度
= 各指标权重 × 各指标数据置信度
```

前台可表达为：

```text
综合推荐分 78
证据覆盖度 65% · 仍缺 TTFT
```

低证据覆盖度线路可以参与比较，但不能标记为“强推荐”或“已充分验证”。

## 8. 指标规划

### 8.1 模型质量：25%

模型质量必须跟随具体模型与分组，不采用站点整体平均。

| 子项 | 综合权重 | 说明 |
| --- | ---: | --- |
| 模型真实性与身份一致性 | 10% | 是否为目标模型，是否暗调其他模型或路由 |
| 能力完整度与降质检测 | 10% | 推理、工具调用、视觉、上下文等能力是否符合预期 |
| 协议与输出完整性 | 5% | 流式响应、工具调用格式、结束原因和返回结构是否正常 |

产品规则：

- 明确假模型、错误路由：退出对应模型线路的综合推荐。
- 疑似降质或需要复核：允许展示，但限制最高推荐等级。
- 未检测：使用中性先验并降低证据覆盖度。
- 检测通过：根据检测分、报告数量、检测时间计算置信度。
- 检测结果随时间衰减，不能永久有效。

### 8.2 价格：25%

价格只在同一个标准模型的候选线路中比较，不使用其他模型的最低价格代表当前线路。

建议继续采用带异常值保护的对数归一化：

- 同模型候选中的低价区间接近高分。
- 高价区间接近低分。
- 极端低价不会无限放大优势。
- 价格缺失的线路不进入该模型综合推荐。

当线路输入、输出和缓存价格拆分明显时，后续使用统一标准负载折算成本，避免只看输入便宜但输出昂贵。

### 8.3 响应速度：20%

流式模型的响应体验不能只看一个平均延迟。目标结构为：

| 子项 | 综合权重 | 说明 |
| --- | ---: | --- |
| TTFT P50 | 12% | 用户大多数请求的首 Token 体感 |
| TTFT P95 | 4% | 偶发慢请求与尾部延迟 |
| TPS 或标准输出总耗时 | 4% | 首 Token 后的持续生成速度 |

TTFT 统一定义为：

```text
请求正式转发给上游
→ 客户端收到第一段有效 SSE 内容
```

比较边界：

- 只在同一个标准模型内比较。
- 公开统计必须披露窗口、请求样本量、更新时间和探测请求类型。
- 上下文大小、推理等级和图片输入明显不同时，不作为同一批可比样本。
- 如果总耗时没有固定输出长度，只展示，不进入评分；优先使用 TPS。
- 当前已有的普通延迟字段不冒充 TTFT。

### 8.4 稳定性：20%

| 子项 | 综合权重 | 说明 |
| --- | ---: | --- |
| 去重后的最近 60 次有效成功率 | 12% | 反映当前故障和短期变化 |
| 近 7 日成功率 | 8% | 反映长期稳定基线 |

两项都使用带样本置信度的保守成功率，例如 Wilson 下界。

“有效成功”应逐步统一为：

- 请求返回成功。
- 流式响应正常结束。
- 返回内容非空。
- 没有明显协议错误。

推荐门槛建议：

- 成功率明显低于 90%：不进入综合推荐区。
- 90%–95%：允许展示，但标记稳定性风险。
- 95% 以上：正常参与推荐。
- 数据超过 7 天未更新：退出综合推荐。
- 相同公开监测事件在多个模型或分组重复出现时，只按真实事件数量计入置信度。

### 8.5 缓存效率：10%

缓存命中率采用已确认的上游口径：

```text
缓存命中率
= 缓存读取 Token
÷（输入 Token + 缓存创建 Token + 缓存读取 Token）
```

产品规则：

- 使用同模型、同分组、同统计窗口的数据。
- 优先采用近 7 日窗口。
- Token 样本量采用对数封顶，只用于判断置信度，不线性增加得分。
- Claude 与 GPT 等不同模型不直接用缓存率互相比较。
- 没有缓存数据时使用中性先验并标记未披露。
- 模型或线路不适用缓存时标记“不适用”，不视为低分。

第一阶段可直接使用缓存命中率作为缓存效率分。后续如果公开数据足够，可在 10% 内部拆成：

- 缓存命中覆盖：6%。
- 缓存带来的实际成本节省：4%。

此时价格维度使用未考虑缓存的基础价格，缓存维度表达增量节省，避免重复计算。

## 9. 分组线路与向上汇总

### 9.1 用户可以选择分组

- 分组分别评分。
- 站点在具体模型页面展示“推荐分组”。
- 必须同时展示分组名称、价格、稳定性、缓存、质量状态和更新时间。
- 用户可展开查看同站点其他分组，不能把最佳分组伪装成整站表现。

### 9.2 用户不能选择分组，由站点自动路由

- 有公开流量比例时按实际流量加权。
- 没有流量比例时，不直接取最好分组；采用较保守的分组汇总结果。
- 分组差距明显时显示“线路差异较大”。
- 混合路由的检测和监测结论必须明确证据边界。

### 9.3 模型家族页面

- 由家族内代表模型的线路分向上汇总。
- 不取家族中最便宜的任意模型代表整个家族。
- 展示模型覆盖率和样本覆盖情况。

### 9.4 “全部”页面

“全部”页面暂时保留为待确认产品决策，候选方向包括：

1. 保留一个综合推荐榜，但使用固定模型篮子和覆盖门槛。
2. 改成省钱、稳定、低延迟、缓存、已检测等场景推荐。
3. 仅做站点总览，把严格排序留在具体模型和模型家族页面。

无论最终选择哪种展示方式，底层统一复用线路分，不再重新发明一套站点总分。

## 10. 核心用户旅程

```text
进入中转 API
→ 选择模型家族或具体模型
→ 查看符合条件的站点与分组线路
→ 按综合推荐、价格或稳定性排序
→ 查看综合分、证据覆盖度和推荐理由
→ 展开同站其他分组进行比较
→ 进入详情核对来源、缓存、检测和更新时间
→ 回原站小额试用
```

产品需要让用户在 30 秒内完成：

```text
锁定 2–3 条值得小额试用的目标模型线路
```

## 12. 关键状态与边界

| 状态 | 产品处理 |
| --- | --- |
| 没有 TTFT 数据 | 使用中性先验，显示“暂缺速度数据”，不按 0 分惩罚 |
| 没有模型检测 | 使用中性先验，显示“待检测”，不能标记强推荐 |
| 缓存未披露 | 使用中性先验并降低证据覆盖度 |
| 数据样本少 | 指标向中位值收缩，显示样本不足 |
| 数据过期 | 超过 7 天退出综合推荐，继续保留资料展示 |
| 明确假模型 | 对应模型线路退出推荐，保留检测证据和风险说明 |
| 分组差异大 | 展示最佳与其他分组差距，自动路由采用保守汇总 |
| 商业合作站点 | 明确披露合作关系，但自然推荐分不受影响 |

## 13. 分阶段路线

### 阶段 0：中立综合排序基线，已完成

- 移除商业关系、站点系统、资料完整度和主观风险标签的性能加分。
- 接入成本、近 60 次、近 7 日、缓存和模型检测预留权重。
- 建立无倍率、无稳定性、数据过期的推荐准入规则。
- 增加排序回归测试。
- 生产证据：commit `5fb509b`，Cloudflare Actions `29267043533`。

### 阶段 1：线路级评分与证据覆盖度，待确认

- 把排序对象收敛为站点 × 模型 × 分组线路。
- 建立公开监测事件去重。
- 缓存严格跟随分组。
- 增加中性先验和证据覆盖度。
- 前台展示推荐分组和主要推荐理由。

### 阶段 2：模型质量接入，待后续实施

- 接入真实性、能力完整度和协议检测。
- 建立质量失败退出规则和疑似降质上限。
- 处理报告复用、报告时效和多次检测置信度。

### 阶段 3：TTFT 与输出速度，待公开口径成熟

- 统一 TTFT 起止定义。
- 接入 P50、P95、请求样本量和更新时间。
- 接入 TPS 或固定输出长度总耗时。
- 按同模型、可比探测请求归一化。

### 阶段 4：“全部”页面决策与权重校准，待讨论

- 确认单一总榜、场景推荐或站点总览方向。
- 用历史公开数据回放不同权重下的排名变化。
- 检查极端低价、样本膨胀、缺失数据和单次检测是否造成跳榜。
- 根据用户点击、详情查看和反馈验证推荐是否真正帮助决策，但不把商业转化直接作为自然排序分。

## 14. 成功标准

| 指标 | 说明 | 第一阶段目标 |
| --- | --- | --- |
| 可解释性 | 用户是否能看懂推荐原因 | 每条推荐至少展示 2–3 个主要理由和缺失项 |
| 可比性 | 是否比较同模型、同线路数据 | 具体模型排序不借用其他模型最低价或缓存值 |
| 中立性 | 商业字段是否影响自然排序 | 商业关系变化不改变性能分 |
| 样本可信度 | 重复样本是否被放大 | 同一公开监测事件只计一次置信度 |
| 缺失公平性 | 未披露是否等同真实低分 | 缺失数据使用中性先验并降低证据覆盖度 |
| 用户决策效率 | 用户能否快速缩小范围 | 30 秒内筛出 2–3 条值得小额试用的线路 |
| 排名稳定性 | 小幅数据变化是否导致剧烈跳榜 | 对单次样本、单次检测和极端低价具有保护 |
