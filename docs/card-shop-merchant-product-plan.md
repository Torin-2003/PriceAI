# PriceAI 卡网商家板块产品规划

> 文档状态：草案
> 更新日期：2026-06-24
> 面向模块：卡网订阅、渠道报价、商家观察
> 目标：把分散在商品报价里的来源信息，沉淀成可浏览、可排序、可核验的“卡网商家”视角。

## 1. 背景

PriceAI 当前卡网订阅模块已经有两个主要视角：

| 视角 | 用户问题 | 当前状态 |
| --- | --- | --- |
| 标准商品 | 我想买什么？哪个商品当前最低？ | 已上线 |
| 全部报价 | 具体有哪些原始报价？ | 已上线 |

用户反馈里出现了第三个问题：我能不能先看商家本身。尤其是卡网渠道里，新店、短期店、无售后店、商品描述不清楚的店，会让用户不敢直接下单。单个商品价格便宜，不代表这个店铺长期稳定、更新及时、售后入口清楚。

因此建议新增第三个视角：

| 视角 | 用户问题 | 建议入口 |
| --- | --- | --- |
| 卡网商家 | 哪些店长期有货、覆盖多、更新勤、风险反馈少？ | 卡网订阅模块里的一级视图 |

该板块不做“担保榜”或“信誉榜”，而做“PriceAI 观察到的商家资料和经营信号”。页面要持续强调：PriceAI 不卖货、不收款、不替任何商家背书。

## 2. 链动小铺公开字段调研结论

本次抽样检查了 `pay.ldxp.cn` 链动小铺类公开接口，重点看 `Shop/info`、`categoryList`、`goodsList` 是否能提供店铺时长、销量、商品数等字段。

结论：链动小铺公开接口里可以拿到店铺创建时间和店铺累计销量。

### 2.1 `Shop/info` 可用字段

抽样店铺的 `Shop/info` 响应中出现以下字段：

| 字段 | 含义判断 | 前台建议 |
| --- | --- | --- |
| `link` | 店铺入口 | 可展示 |
| `create_time` | 店铺创建时间，Unix 秒级时间戳 | 可转为“开店时间 / 店铺时长” |
| `nickname` | 店铺名 | 可展示 |
| `avatar` | 店铺头像 | 可展示，但可选 |
| `description` | 店铺说明 | 详情页可摘要，避免直接塞长文本 |
| `sell_count` | 店铺累计销量或成交数 | 可展示为“平台公开销量”，需标注口径 |
| `goods_count` | 商品总数 | 可展示 |
| `card_count` / `resource_count` / `equity_count` | 分类型商品数 | 可用于品类画像 |
| `deposit_money` | 保证金/押金类字段 | 可作为待核验字段，不应直接解释成担保 |
| `contact_qq` / `contact_wechat` / `contact_mobile` | 公开联系方式 | 前台优先展示“售后入口已公开”，不建议列表页直接暴露号码 |
| `custom_status` / `custom_status_msg` | 店铺自定义营业状态 | 可用于状态提示 |
| `params.show_sell_count` | 是否展示销量的配置 | 建议尊重该开关，决定是否前台展示销量 |
| `params.show_deposit_money` | 是否展示保证金/押金的配置 | 建议尊重该开关，决定是否前台展示 |
| `auth_status` | 认证状态，语义需继续确认 | 暂不前台解释 |

抽样换算：

| 样本 | `create_time` 换算 | 说明 |
| --- | --- | --- |
| LToken AI 小铺 | 2026-04-10 | 响应包含 `create_time`、`sell_count`、`goods_count` |
| Rick 的杂货铺 | 2026-04-10 | 响应包含 `create_time`、`sell_count`、`goods_count` |
| 咕咕 Ai 源头号商 | 2026-03-31 | 响应包含 `create_time`、`sell_count`、`goods_count` |
| 大模型 API 服务商 | 2026-04-13 | 响应包含 `create_time`、`sell_count`、`goods_count` |

### 2.2 `goodsList` 可用字段

`goodsList` 的商品项中出现以下字段：

| 字段 | 含义判断 | 前台建议 |
| --- | --- | --- |
| `link` | 商品链接 | 已可用 |
| `goods_key` | 商品 key | 内部归一用 |
| `name` | 商品标题 | 已可用 |
| `price` / `market_price` | 商品展示价 | 已可用，结算价仍应以当前采集器的实付价逻辑为准 |
| `create_time` | 商品创建时间 | 可做“商品上架时间”或“老商品/新品”信号 |
| `category.name` | 分类名 | 已可用于标签 |
| `extend.stock_count` | 库存数量 | 已可用 |
| `extend.send_order` | 自动/人工发货信号 | 已可用 |

本次未在 `goodsList` 列表项中确认单个商品销量字段。商品级销量如果后续确实需要，可以低频探索商品详情接口，但不建议 V1 为了商品销量对每个商品额外请求详情页。

### 2.3 对现有采集器的影响

当前 `shopApi` 采集器已经读取：

- 店铺昵称：`Shop/info.data.nickname`
- 店铺入口：`Shop/info.data.link`
- 分类列表：`categoryList`
- 商品列表：`goodsList`
- 商品库存：`extend.stock_count`
- 商品实付价：`getGoodsPrice`

但当前没有把 `create_time`、`sell_count`、`goods_count`、`deposit_money` 等店铺画像字段沉淀进 PriceAI 的结构化数据。

## 3. 产品定位

### 3.1 一句话定义

卡网商家板块是 PriceAI 面向第三方卡网渠道的商家观察页，帮助用户在购买前按店铺维度查看覆盖商品、库存状态、更新频率、公开销量、店铺时长、售后入口和风险反馈。

### 3.2 产品边界

本板块做：

- 聚合 PriceAI 已采集到的商家和店铺。
- 展示可公开核验的商家经营信号。
- 按商品覆盖、有货、更新、销量、店龄、风险反馈等维度排序。
- 让用户从商家进入该店在 PriceAI 收录的商品和报价。

本板块不做：

- 不对商家做安全担保。
- 不把销量、店龄直接等同于可信。
- 不参与交易、收款、仲裁或售后。
- 不绕过登录、验证码、风控或非公开后台。

## 4. 目标用户和场景

| 用户 | 场景 | 需求 |
| --- | --- | --- |
| 普通买家 | 准备买 ChatGPT Plus、Claude、Gemini、邮箱、接码等商品 | 想知道某个店是不是新店、是否长期有货、是否有售后入口 |
| 进阶买家 | 已经知道多个卡网店铺 | 想比较哪些店覆盖多、更新勤、低价命中多 |
| 风险敏感用户 | 准备买大额代充、Pro、Team、API 额度 | 想先看店铺观察时间、公开销量、风险反馈 |
| PriceAI 运营者 | 审核渠道和定位异常报价 | 想快速发现长期失败、商品少、风险多、疑似重复或异常店铺 |

## 5. 页面结构

### 5.1 入口

建议在卡网订阅模块的视图切换中加入：

```text
标准商品 / 全部报价 / 卡网商家
```

不要把“卡网商家”放成营销页，也不要单独做大段介绍。它应该像现有榜单一样，直接进入筛选和表格。

### 5.2 筛选设计

筛选先围绕用户真的会比较的几个问题，不要一次做成复杂后台。

| 筛选 | 选项建议 | 数据来源 | 说明 |
| --- | --- | --- | --- |
| 采集来源 | 链动小铺 / 独角数卡 / Kami / 其他 | `sources.collector_kind`、采集器识别结果 | 这是商家页最重要的一级筛选。链动小铺对应内部 `shopApi`，独角数卡对应 `dujiao`，Kami 对应 `kami`。 |
| 商品平台 | ChatGPT / Claude / Gemini / Grok / 邮箱 / 接码 / API/CDK / 其他 | 标准商品、分类、标题标签 | 和现有卡网订阅平台筛选保持一致。 |
| 库存状态 | 有货为主 / 全部有货 / 全部缺货 / 全部 | `raw_offers.status`、`effective_status` 聚合 | 默认建议偏向有货商家。 |
| 价格表现 | 有最低价命中 / 有质保最低价命中 / 两者都有 | 标准商品最低价、质保最低价反向聚合 | 让用户快速找到“便宜”和“质保便宜”的店。 |
| 售后信号 | 有平台售后机制 / 有售后入口 / 有售后群 / 待补充 | 链动平台机制、公开页面、商家补充、用户反馈 | V1 先做状态，不直接判断售后好坏。 |
| 采集状态 | 正常 / 重试中 / 失败 / 近期未更新 | `sources.health_status`、`last_success_at`、失败次数 | 主要给用户解释数据是否新鲜。 |
| 商家观察 | 有平台店龄 / 有公开销量 / PriceAI 观察较久 / 新观察 | 链动公开字段、PriceAI 聚合 | 链动优先补齐，其他平台暂用 PriceAI 观察天数。 |
| 风险反馈 | 无待核验 / 有待核验 / 有已核验风险 / 商家已回应 | 用户反馈、审核结果、商家回应 | 避免把未核验反馈做成定罪式标签。 |

### 5.3 列表页字段

MVP 列表建议字段：

| 字段 | 展示名 | 数据来源 | V1 可行性 |
| --- | --- | --- | --- |
| 店铺名 | 商家/店铺 | `source_name`、`source_store_name`、链动 `nickname` | 可做 |
| 采集来源 | 来源 | `sources.collector_kind` | 可做，建议显示为链动小铺、独角数卡、Kami、其他 |
| 店铺入口 | 原站 | `sources.entry_url`、链动 `link` | 可做 |
| 商品数 | 商品 | `raw_offers` 聚合、链动 `goods_count` | 可做 |
| 有货数 | 有货 | `status` / `effective_status` 聚合 | 可做 |
| 覆盖品类 | 覆盖 | 标准商品、平台、分类聚合 | 可做 |
| 最低价命中 | 最低价命中 | 对当前标准商品最低报价做来源聚合 | 可做 |
| 质保最低价命中 | 质保最低价命中 | 对当前标准商品质保最低报价做来源聚合 | 可做，现有商品口径已有 `warrantyLowestOffer` |
| 最近更新 | 更新 | `verified_at`、`last_seen_at`、`source_updated_at` | 可做 |
| 采集状态 | 状态 | `health_status`、`last_success_at`、失败次数 | 可做 |
| PriceAI 观察天数 | 观察 | 最早 `raw_offers.created_at` / `sources.created_at` | 可做，但需明确口径 |
| 店铺时长 | 店龄 | 链动 `create_time` 等平台公开字段 | 链动可做，其他平台待适配 |
| 平台公开销量 | 销量 | 链动 `sell_count` | 链动先试运行，后续按反馈调整口径 |
| 平台售后机制 | 售后机制 | 链动平台机制、采集来源 | 链动可展示“有平台订单/售后机制”，但不等于 PriceAI 担保 |
| 售后入口状态 | 售后入口 | 链动联系方式、原站描述、商家补充 | 可做状态，不直接展示联系方式 |
| 售后群数量 | 售后群 | 商家补充、页面描述抽取 | 规划字段，V1 可先预留 |
| 售后质量 | 售后质量 | 用户反馈、商家回应、后续商家自填 | 规划字段，不建议 V1 强排序 |
| 风险反馈 | 风险 | `offer_feedback` / `riskFeedback` 聚合 | 可做 |

### 5.4 详情页字段

商家详情页建议按“先看概况，再看证据”的顺序分区：

1. 商家概览：店铺名、头像、采集来源、原站入口、PriceAI 观察天数、店铺时长、平台公开销量、商品数、有货数、最近更新。
2. 采集与平台机制：采集来源、采集状态、最近成功、连续失败、平台机制说明。链动小铺可单独标注“有平台订单/售后机制”，但文案要避免写成担保。
3. 价格贡献：最低价命中数、质保最低价命中数、代表低价商品、代表质保低价商品。
4. 商品覆盖：按 ChatGPT、Claude、Gemini、Grok、邮箱、接码、API/CDK 等聚合，展示该店覆盖哪些标准商品。
5. 库存与更新：有货商品、缺货商品、近期新增商品、长期有货商品、最后确认时间。
6. 售后与反馈：售后入口状态、售后群数量、售后质量记录、用户反馈摘要、商家回应。联系方式不在 PriceAI 列表页直接展示，用户需要去原站查看。
7. 公开经营信息：店铺说明摘要、平台公开销量、公开商品数、店铺创建时间。`deposit_money`、`auth_status` 先放后台观察，不在前台强解释。
8. 风险提示：新观察、长时间未更新、全部缺货、待核验风险、已核验风险、商家已回应。
9. 原站跳转：进入店铺、进入该店具体商品。

## 6. 排序设计

### 6.1 默认排序：综合观察

默认排序不应该叫“信誉排序”。建议叫“综合观察”，由以下信号组成，并分阶段加入权重。

| 信号 | 初始权重建议 | 正向 | 负向 |
| --- | --- | --- | --- |
| 有货商品数 | 高 | 有货越多越靠前 | 全部缺货降权 |
| 更新新鲜度 | 高 | 最近确认越新越靠前 | 长时间未成功采集降权 |
| 采集稳定性 | 高 | 最近成功、失败少 | 连续失败、风控频繁降权 |
| 商品覆盖 | 中高 | 覆盖标准商品越多越靠前 | 单一商品且样本少中性 |
| 最低价命中 | 中高 | 经常是当前最低价加分 | 不应压过风险信号 |
| 质保最低价命中 | 中高 | 质保商品也有价格优势加分 | 样本少时只做轻加分 |
| 平台售后机制 | 中 | 链动小铺这类有平台订单/售后机制的来源加分 | 不能替代真实售后反馈 |
| 观察时间 | 中 | PriceAI 观察时间越长加分 | 新店不直接惩罚，但标注“新观察” |
| 店铺时长 | 中 | 平台公开店龄越长加分 | 缺失不直接惩罚 |
| 平台公开销量 | 中低 | 销量高加分 | 只做轻权重，避免刷量误导 |
| 售后群数量 | 低，先观察 | 售后群越完整越加分 | 自填或抽取字段不宜强排序 |
| 售后质量 | 低到中，后置 | 用户反馈好、回应及时加分 | 已核验售后问题降权 |
| 风险反馈 | 高 | 风险少、已回应加分 | 已核验风险、集中投诉降权 |

### 6.2 分阶段权重策略

| 阶段 | 排序依据 | 说明 |
| --- | --- | --- |
| V1 客观数据 | 有货数、更新、采集稳定性、覆盖、最低价命中、质保最低价命中、风险反馈 | 先不依赖商家自填，避免刚上线就被主观字段影响。 |
| V2 链动画像 | 店铺时长、平台公开销量、链动平台售后机制 | 链动小铺公开字段先试运行，用反馈校准权重。 |
| V3 商家反馈 | 售后入口、售后群数量、售后质量、商家回应 | 商家自填字段只做辅助，需配合用户反馈和审核。 |

### 6.3 可选排序

| 排序 | 适用用户问题 |
| --- | --- |
| 综合观察 | 默认推荐哪些店更值得先看？ |
| 有货最多 | 哪些店现在供货最多？ |
| 最低价命中 | 哪些店经常给当前最低价？ |
| 质保最低价命中 | 哪些店经常给当前质保最低价？ |
| 更新最近 | 哪些店最近确认过？ |
| 覆盖最多 | 哪些店品类最全？ |
| 店龄较长 | 哪些店开店时间更久？ |
| 观察时间长 | 哪些店被 PriceAI 跟踪更久？ |
| 销量高 | 哪些店平台公开销量更高？ |
| 风险较少 | 哪些店用户风险反馈少？ |

## 7. 字段补全策略

### 7.1 第一批可直接补全

这些字段可以基于现有数据或链动公开接口补齐：

| 字段 | 补全方式 |
| --- | --- |
| 采集来源 | 使用 `sources.collector_kind`，前台归类为链动小铺、独角数卡、Kami、其他 |
| PriceAI 观察天数 | 聚合该来源最早报价创建时间、最早采集成功时间或 `sources.created_at` |
| 商品数 / 有货数 / 缺货数 | 从 `raw_offers` 聚合 |
| 覆盖标准商品数 | 从 `raw_offers.canonical_product_id` 聚合 |
| 覆盖平台/品类 | 从标准商品、分类标签和报价标题聚合 |
| 最低价命中数 | 在标准商品最低价计算中反向聚合来源 |
| 质保最低价命中数 | 在标准商品质保最低价计算中反向聚合来源 |
| 最近更新 | `verified_at`、`last_seen_at`、`source_updated_at` 取最大值 |
| 采集健康 | `sources.health_status`、`last_success_at`、`consecutive_failures` |
| 风险反馈 | 聚合来源级和报价级用户反馈 |
| 店铺创建时间 | 链动 `Shop/info.create_time` |
| 平台公开销量 | 链动 `Shop/info.sell_count` |
| 公开商品数 | 链动 `Shop/info.goods_count`、分类商品数 |
| 商品上架时间 | 链动 `goodsList.create_time` |

### 7.2 需要谨慎解释的字段

| 字段 | 风险 | 建议 |
| --- | --- | --- |
| 销量 | 可能被刷，也可能是平台累计口径 | 先试运行，前台标注“平台公开销量”，不写“真实销量” |
| 店龄 | 链动可拿，但其他平台未必有 | 链动展示“开店时间”，其他显示“PriceAI 观察天数” |
| 链动平台售后机制 | 平台有订单/售后机制，不等于最终售后质量好 | 可作为优先推荐信号，但文案只写“有平台订单/售后机制” |
| 保证金/押金 | 不等于 PriceAI 担保，也不一定能覆盖用户损失 | 先后台观察，暂不前台解释 |
| 售后入口 | 有联系方式不代表售后有效 | 列表只展示“已公开/未公开/待确认”，不直接展示联系方式 |
| 售后群数量 | 可自填、可变化，也可能被包装 | 先做规划字段，后续配合商家补充和人工审核 |
| 售后质量 | 主观性强，容易被刷 | 初期只沉淀用户反馈和商家回应，不做强排名 |
| 认证状态 | `auth_status` 语义需确认 | 暂不前台展示 |
| 登录地区 | 与购买判断弱相关，也可能引发误解 | 不建议前台展示 |

### 7.3 需要商家或后续标准接口补全

| 字段 | 补全方式 |
| --- | --- |
| 售后群数量 | 商家补充 + 页面描述抽取 + 人工审核 |
| 售后质量 | 用户反馈、工单处理记录、商家回应、人工审核 |
| 售后响应时长 | 商家提供 + 用户反馈验证 |
| 退款处理规则 | 商家提供公开说明 |
| 官方开店主体 | 商家认领后提供 |
| 发票能力 | 商家认领后提供 |
| 近 30 日销量 | 商家提供摘要或平台公开统计 |
| 平台认证含义 | 与平台字段确认或通过更多样本校验 |

## 8. 数据口径建议

### 8.1 前台命名

| 原始概念 | 前台建议名 | 说明 |
| --- | --- | --- |
| `shopApi` | 链动小铺 | 面向用户用平台名，内部保留采集器类型 |
| `dujiao` | 独角数卡 | 归入采集来源筛选 |
| `kami` | Kami | 归入采集来源筛选 |
| 其他采集器 | 其他 | 避免列表筛选过碎 |
| 链动 `create_time` | 开店时间 | 仅当平台公开字段存在时展示 |
| 链动 `sell_count` | 平台公开销量 | 不写“真实销量” |
| PriceAI 最早采集 | PriceAI 观察天数 | 不冒充店铺真实年龄 |
| `health_status` | 采集状态 | 表示 PriceAI 能不能读到，不表示商家是否可靠 |
| 链动平台机制 | 平台订单/售后机制 | 不写“官方担保”或“PriceAI 推荐担保” |
| 风险反馈 | 用户反馈待核验 | 未核验前不做定罪式表达 |

### 8.2 列表文案

建议在页面底部或表头附近放一行短说明：

```text
卡网商家页展示 PriceAI 从公开页面和公开接口观察到的店铺信号。销量、店龄、商品数、售后入口等字段以原平台公开口径或商家补充信息为准，不构成购买担保。
```

## 9. 实施阶段

### 阶段 1：卡网商家 MVP

目标：不新增商家合作流程，先基于现有报价数据做商家聚合页。

范围：

- 新增“卡网商家”视图。
- 展示店铺名、采集来源、入口、商品数、有货数、覆盖商品、最近更新、采集状态、PriceAI 观察天数。
- 展示最低价命中和质保最低价命中。
- 支持综合观察、有货最多、更新最近、覆盖最多、最低价命中、质保最低价命中排序。
- 支持按采集来源、平台/品类、采集状态、有货状态、价格表现筛选。

### 阶段 2：链动公开画像增强

目标：把链动小铺公开字段沉淀为商家画像。

范围：

- 在 `shopApi` 采集过程中保存 `create_time`、`sell_count`、`goods_count`、商品 `create_time` 等字段。
- 列表展示“开店时间”“平台公开销量”“公开商品数”。
- 详情页展示店铺描述摘要、售后入口状态、平台订单/售后机制。
- 尊重 `show_sell_count`、`show_deposit_money` 等展示开关。
- `sell_count` 先按“平台公开销量”试运行，后续结合用户和商家反馈校准口径。

### 阶段 3：售后和反馈信号沉淀

目标：先把售后相关字段沉淀起来，但不急着做强排名。

范围：

- 售后入口状态：已公开、未公开、待确认。
- 售后群数量：先作为规划字段，可由商家补充或页面描述抽取。
- 售后质量：先由用户反馈、审核结果、商家回应沉淀。
- 用户反馈挂到商家和商品两层。
- 商家回应可以前台披露，但需标注来源。

### 阶段 4：商家认领和多平台统一

目标：在链动小铺跑通后，再扩展到独角数卡、Kami 和其他平台。

范围：

- 商家认领入口，登录体系后置。
- 商家补充售后规则、发票能力、主体类型、联系方式、优惠说明。
- 为不同采集器定义商家画像适配层。
- 没有公开店龄的平台统一使用“PriceAI 观察天数”。
- 没有公开销量的平台不展示销量，不用人工猜。

## 10. 成功标准

| 指标 | 目标 |
| --- | --- |
| 商家页可用性 | 用户能从商家页进入原店和该店商品 |
| 数据解释清晰 | 用户能区分“平台公开字段”和“PriceAI 观察字段” |
| 风险表达克制 | 页面没有把销量、店龄包装成担保 |
| 来源筛选清晰 | 用户能按链动小铺、独角数卡、Kami、其他筛选商家 |
| 价格表现可解释 | 用户能同时看到最低价命中和质保最低价命中 |
| 售后信号可沉淀 | 售后入口、售后群数量、售后质量可以逐步补全 |
| 采集负担可控 | 链动小铺仍遵守低频、串行、冷却策略 |
| 运营效率提升 | 后台能更快发现异常店铺、重复店铺、长期失败店铺 |

## 11. 本阶段边界

- 商家登录体系：商家认领和登录后置，当前先把公开数据和 PriceAI 观察跑通。
- 非链动平台店龄：独角数卡、Kami 等平台后续逐个确认，不影响链动小铺优先落地。
- 直接展示联系方式：前台列表不直接展示 QQ、微信、手机号，只展示售后入口状态和原站跳转。

## 12. 待确认问题

1. 链动 `sell_count` 先按“平台公开销量”试运行，后续根据样本和用户反馈调整解释。
2. 链动 `deposit_money` 和 `auth_status` 的准确语义不清楚，先后台观察，不做前台强解释。
3. 售后群数量、售后质量需要确定采集方式和审核方式，不能完全依赖商家自填。
4. 独角数卡、Kami 和其他平台的店龄、销量、售后字段后续逐个适配。
5. 商家认领和登录体系后续再规划，当前不是 MVP 前置条件。
