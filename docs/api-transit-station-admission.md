# PriceAI API 中转站收录与合作说明

> 文档状态：草案
> 面向对象：API 中转站站长、上游渠道商、系统服务商
> 更新日期：2026-06-24

PriceAI 做的是 API 中转站的公开比价、资料核验和可用性观察。我们不售卖 API，也不替任何商家做担保。收录和合作的前提是：站点愿意把用户购买前必须知道的价格、倍率、模型分组、上游类型和可用性信息公开出来。

## 1. 收录原则

PriceAI 后续会把 API 中转站分成几个状态：

| 状态 | 含义 |
| --- | --- |
| 正式上架 | 满足公开资料、分组披露、价格口径和监测页要求，可以进入站点榜和详情页。 |
| 待核验 | 基础资料存在，但部分分组、上游来源、监测样本或价格口径还不完整。 |
| 仅后台线索 | 只有站点线索或私聊资料，暂不对外展示。 |
| 暂不收录 | 缺少公开价格和监测信息，或拒绝披露关键来源类型。 |

正式上架不等于 PriceAI 推荐。榜单排序仍以价格、可用性、样本数、资料完整度和用户反馈为主；商业合作会单独标识，不直接影响客观排序。

## 2. 正式上架的基础门槛

正式上架至少需要满足以下条件：

1. 提供可访问的官网地址。
2. 官网必须有公开价格页或公开只读数据接口。
3. 公开充值倍率，例如 `1 元 = 1 额度`、`1 元 = 1.2 额度`。
4. 公开最低充值、余额有效期、退款规则和售后入口。
5. 公开所有对外展示的模型分组倍率。
6. 每个模型分组必须说明上游来源、账号池或渠道类型。
7. 每个要展示的模型分组必须有可用性监测页或可被 PriceAI 读取的监测数据。
8. 每个模型或模型分组需要有可对照的模型基准价。
9. 公开基础限流口径，例如并发数、RPM、TPM、单 Key 限制或账号级限制。
10. 提供 PriceAI 专属优惠码 `PRICEAI`，具体优惠政策由站点自行设定。

API Base URL 不是强制提交项。PriceAI 当前更关注用户购买前能否公开核验价格、倍率、分组来源和可用性；如后续做真实调用监测，再单独向站点申请专用测试 Key 或测试账号。

## 3. 模型分组披露要求

站点不能只写“高速池”“稳定池”“高级池”这类模糊名称。每个模型分组都需要说明它实际代表什么。

每个分组建议至少公开：

| 字段 | 说明 |
| --- | --- |
| 分组名称 | 站内展示的分组名。 |
| 覆盖模型 | 该分组可用的模型列表或标准模型映射。 |
| 分组倍率 | 该分组相对基准价或站内计费的倍率。 |
| 模型基准价 | 用于计算倍率的公开基准价来源。 |
| 上游类型 | 官方 API、云厂商、订阅转 API、逆向、二级分销、混池等。 |
| 账号池类型 | Free、Plus、Pro、Max、Team、Enterprise、官方 API、混池等。 |
| 可用性监测 | 该分组对应的公开监测页或监测数据入口。 |
| 限流口径 | 并发数、RPM、TPM、单 Key 限制、账号级限制或是否未承诺。 |
| 更新时间 | 价格、倍率和监测数据最后更新时间。 |

### 3.1 Claude 分组

Claude 相关分组需要说明：

- 是官方 API、云厂商、订阅转 API、逆向、二级分销，还是混池。
- 如果是订阅转 API 或逆向，必须明确标注，不能包装成官方 API。
- 如果来自账号池，需要说明是 Pro、Max、Team、Enterprise 或其他池子。
- 如果同一个分组会在多个上游之间切换，需要标注为混池，并说明可能影响稳定性和返回模型一致性。

### 3.2 GPT 分组

GPT 相关分组需要说明：

- 是官方 API、云厂商、订阅转 API、逆向、二级分销，还是混池。
- 如果来自账号池，需要说明是 Free、Plus、Pro、Team、Enterprise 或其他池子。
- 如果不同模型走不同池子，需要按模型或分组拆开，不能只写一个全站 GPT 倍率。
- 如果存在普通号、Plus 号、Pro 号、Team 号混用，需要明确标注混池。

## 4. 模型基准价要求

模型基准价用于让不同站点的倍率可以横向比较。站点需要说明：

- 基准价来源，例如官方公开 API 价格、云厂商价格页或站内自定义基准价。
- 基准价单位，例如每 1M tokens 输入价格、输出价格、缓存读价格、缓存写价格。
- 基准价更新时间。
- 如果站点使用自己的内部基准价，需要说明它和官方公开价格之间的关系。

PriceAI 会优先采用公开、可复查的基准价。缺少基准价时，该分组可以进入待核验，但不适合进入正式价格对比。

## 5. 限流与并发披露要求

并发数、RPM 和 TPM 会直接影响用户在 Claude Code、Cursor、自动化脚本、批处理任务里的真实体验，因此属于正式上架的基础披露项。

基础门槛不是要求站点达到某个固定并发或 RPM，而是要求站点把限制说清楚。站点至少需要说明：

- 每个模型分组的并发限制。
- 每个模型分组的 RPM 或 RPS 限制。
- 如有 TPM、TPD、日额度或小时额度限制，也需要说明。
- 限制作用范围：单 API Key、单账号、单 IP、单用户组、全站共享池，还是按模型分组计算。
- 超限后的表现：排队、429、降级、切换上游、临时封禁或其他策略。
- 是否存在高峰期动态限流。

如果站点不承诺固定并发或 RPM，可以写“未承诺固定值”或“按上游实时状态动态调整”，但不能把该分组宣传成无限制或高并发稳定。

更高并发、更高 RPM、压测报告、真实负载曲线，可以作为增值展示或高级监测项，不作为基础上架门槛。

## 6. 缓存计费与缓存命中率

缓存相关信息分成两层：

| 项目 | 建议定位 | 说明 |
| --- | --- | --- |
| 缓存读写价格 | 基础披露项 | 如果站点支持缓存计费或对缓存读写单独计价，需要公开缓存读、缓存写的倍率或价格。 |
| 缓存支持范围 | 基础披露项 | 说明哪些模型、哪些分组支持缓存，是否兼容常见工具。 |
| 缓存命中率 | 高级监测项 | 命中率和用户请求形态强相关，不适合作为所有站点的基础准入门槛。 |

如果站点把“高缓存命中”“缓存更便宜”“适合长上下文复用”作为卖点，就需要提供公开统计或说明口径，例如近 24 小时 / 7 日缓存读写样本、命中率、统计范围和更新时间。

PriceAI 展示缓存命中率时，会标注它是站方报告、授权监测，还是 PriceAI 实测，避免把不同来源的数据混在一起。

## 7. 可用性监测页要求

每个要在 PriceAI 展示的模型分组，都应该有对应的监测页或监测数据。监测页至少需要包含：

- 分组名称。
- 覆盖模型。
- 最近 7 日可用率。
- 最近 7 日样本数。
- 最后检测时间。
- 错误状态或异常说明。
- 基础限流信息，例如并发、RPM、TPM 或动态限流说明。
- 如有条件，提供延迟、首 Token 时间、成功率曲线、近 24 小时状态。

监测页可以是站点已有的公开页面，也可以是后续按 PriceAI 标准输出的公开只读接口。只展示“全站正常”不够，PriceAI 更需要按模型分组看到可用性，因为 Claude、GPT、Gemini 等分组可能来自不同上游。

高级监测可以进一步展示：

- 轻量并发测试下的成功率。
- RPM 区间内的稳定性。
- 限流错误比例。
- 平均延迟、P95 延迟、首 Token 时间。
- 缓存命中率和缓存读写样本。

这些高级监测项可以提高资料完整度和用户参考价值，但第一阶段不作为所有站点的基础上架门槛。

## 8. 优惠码要求

正式合作站点需要提供 PriceAI 专属优惠码：

```text
PRICEAI
```

优惠政策由站点自行设定，可以是：

- 新用户赠送少量额度。
- 首充折扣。
- 充值满额赠送。
- 指定模型分组试用额度。
- 限时活动优惠。

站点需要说明优惠码的有效期、使用限制、最低充值要求、是否可叠加其他活动，以及优惠是否对所有用户公开可用。

## 9. 商务合作方式

PriceAI 可以接受多种合作方式，但所有商业关系都会明确标识，不会包装成客观排名。

### 9.1 AFF / 返佣

如果站点本身有 AFF、邀请码、推广链接或返佣系统，优先使用 AFF 合作。站点需要提供：

- AFF 链接或邀请码。
- 注册数、首充数、充值金额的查看方式。
- 返佣比例。
- 结算周期。
- 最低结算金额。
- 退款、刷量、异常订单处理规则。

### 9.2 月度展示 / 赞助展示

如果站点没有 AFF，也可以走月度展示或赞助展示。展示位置和周期单独沟通，前台会明确标注“广告”“赞助”或类似标识。

赞助展示不等于更可信，也不直接影响 PriceAI 的客观榜单排序。

### 9.3 资料维护 / 入驻合作

站点也可以选择入驻合作，用于维护更完整的站点资料、优惠活动、监测接入和商家回应入口。入驻关系会在前台披露。

## 10. 主体和发票信息

如涉及月度展示、赞助展示、入驻服务或其他商务合作，站点需要补充：

- 主体类型：个人、个体工商户、公司、海外主体。
- 是否能给用户开票。
- 是否能给 PriceAI 提供合作付款所需票据或 Invoice。
- 可开票类型：不开票、电子普通发票、增值税专用发票、自然人代开、海外 Invoice 等。
- 开票最低金额、税点或是否含税、开票周期。

这些信息不会直接决定是否收录，但会影响商务合作方式。

## 11. 后续标准接口方向

PriceAI 后续会提供一份公开数据接口标准，方便 Sub2API、New API、自研系统和其他中转站系统对接。建议方向是站点提供公开只读快照，例如：

```text
/.well-known/ai-transit.json
```

或：

```text
/api/public/transit/v1/snapshot
```

接口只需要暴露白名单字段，例如站点名称、公开价格页、充值倍率、模型分组倍率、模型基准价、上游类型、账号池类型、并发/RPM/TPM 限制、缓存计费口径、可用性监测数据和更新时间。它不应该暴露用户数据、后台配置、密钥、真实账号、内部渠道 ID 或任何敏感信息。

在标准接口正式发布前，站点可以先提供公开价格页和公开监测页。愿意提前接入的站长可以联系 PriceAI，我们会提供字段模板和示例。

## 12. 联系时请提供

联系 PriceAI 时，建议一次性提供以下信息：

- 站点名称。
- 官网地址。
- 公开价格页。
- 公开监测页。
- 当前所有模型分组及倍率。
- 每个分组的上游类型和账号池类型。
- 模型基准价来源。
- 每个分组的并发、RPM、TPM 或动态限流说明。
- 缓存读写价格、缓存支持范围；如有公开统计，也可以提供缓存命中率。
- 最低充值、余额有效期、退款规则、售后入口。
- `PRICEAI` 优惠码方案。
- 是否有 AFF 或返佣系统。
- 是否接受月度展示、赞助展示或入驻合作。
- 主体类型和发票能力。
- 是否愿意后续接入 PriceAI 标准公开接口。

PriceAI 会优先处理资料完整、价格和监测可公开核验、愿意标注上游类型并提供用户优惠的站点。
